Triggers en
SQL Server

Automatizá validaciones, auditorías y controles en la base de datos con disparadores DML, DDL y LOGON, entendiendo cuándo convienen y qué impacto tienen.

¿Qué es un trigger?

Un trigger es un procedimiento almacenado especial que SQL Server ejecuta automáticamente cuando ocurre un evento sobre una tabla, una vista, la estructura de la base o incluso una conexión.

A diferencia de un procedimiento almacenado común, no se invoca manualmente. Se dispara solo cuando ocurre el evento que lo activa, por ejemplo un INSERT, un UPDATE o un DELETE.

Tipos de triggers en SQL Server

SQL Server soporta varios tipos de triggers, según el tipo de evento que queremos observar o interceptar.

Triggers DML

Se activan con INSERT, UPDATE o DELETE sobre tablas o vistas. Son los más usados para validaciones, auditoría y controles automáticos.

Triggers DDL

Se activan ante cambios en la estructura: CREATE, ALTER, DROP. Sirven para registrar modificaciones del esquema o bloquear cambios no autorizados.

Triggers LOGON

Se activan al iniciar sesión en la instancia. Pueden usarse para restringir conexiones por horario, usuario o máquina.

AFTER / FOR vs INSTEAD OF

Característica AFTER / FOR INSTEAD OF
Momento de ejecución Después de que la operación ocurrió En lugar de la operación
¿La fila ya existe? Sí, la operación ya impactó en la tabla No todavía; el trigger decide qué hacer
Uso típico Auditoría, logs, controles posteriores Validaciones previas, vistas actualizables
¿Puede cancelar la operación? Sí, con ROLLBACK Sí, simplemente no ejecutando el cambio
!
Diferencia crítica: en un trigger INSTEAD OF INSERT la fila todavía no está en la tabla al momento del disparo. En un AFTER INSERT, sí.

Las tablas inserted y deleted

Dentro de un trigger DML, SQL Server expone dos tablas temporales especiales con los datos afectados por la operación.

inserted
  • 1
    En un INSERT, contiene las filas nuevas.
  • 2
    En un UPDATE, contiene los valores nuevos.
  • 3
    En un DELETE, queda vacía.
deleted
  • 1
    En un DELETE, contiene las filas eliminadas.
  • 2
    En un UPDATE, contiene los valores anteriores.
  • 3
    En un INSERT, queda vacía.

Resumen por operación

Operación inserted deleted
INSERT Filas nuevas Vacía
DELETE Vacía Filas eliminadas
UPDATE Valores nuevos Valores anteriores
+
Estas tablas permiten comparar estados antes y después del cambio, lo que vuelve a los triggers muy útiles para auditoría y validación.

Ejemplos prácticos

El documento trabaja tres escenarios típicos y muy didácticos: validación, auditoría y control de cambios.

1. Validación con INSTEAD OF INSERT

Evita insertar una marca duplicada. Como es INSTEAD OF, el trigger corre antes de que la fila exista realmente en la tabla.

validacion_marca.sql 
CREATE OR ALTER TRIGGER tr_Marca
ON Marcas
INSTEAD OF INSERT
AS
BEGIN
  SET NOCOUNT ON;

  IF EXISTS (
    SELECT 1
    FROM Marcas m
    INNER JOIN inserted i ON m.Marca = i.Marca
  )
  BEGIN
    RAISERROR('La marca ya existe en la tabla.', 16, 1);
  END
  ELSE
  BEGIN
    INSERT INTO Marcas (Marca, PaisID)
    SELECT Marca, PaisID
    FROM inserted;
  END
END

2. Auditoría con AFTER DELETE

Registra automáticamente en una tabla de auditoría cada producto eliminado, junto con usuario y fecha.

auditoria_delete.sql 
CREATE OR ALTER TRIGGER tr_AuditarEliminacion
ON Productos
AFTER DELETE
AS
BEGIN
  SET NOCOUNT ON;

  INSERT INTO Auditoria (Tabla, Accion, IDAfectado, Usuario, Fecha)
  SELECT 'Productos', 'DELETE', d.ProductoID, SYSTEM_USER, GETDATE()
  FROM deleted d;
END

3. Control de precio con AFTER UPDATE

Compara los valores viejos y nuevos para impedir que un precio se reduzca, aplicando ROLLBACK si detecta la situación.

control_precio.sql 
CREATE OR ALTER TRIGGER tr_ControlPrecio
ON Productos
AFTER UPDATE
AS
BEGIN
  SET NOCOUNT ON;

  IF EXISTS (
    SELECT 1
    FROM inserted i
    INNER JOIN deleted d ON i.ProductoID = d.ProductoID
    WHERE i.Precio < d.Precio
  )
  BEGIN
    ROLLBACK TRANSACTION;
    RAISERROR('No se puede reducir el precio de un producto.', 16, 1);
  END
END
i
En los ejemplos del documento se trabaja correctamente con inserted y deleted como tablas, no como si fueran una sola fila. Eso hace que soporten operaciones que afectan múltiples registros.

Cosas importantes a considerar

Los triggers pueden resolver problemas reales, pero necesitan criterio. Bien usados ayudan mucho; mal usados complican rendimiento y mantenimiento.

  • 1
    Forman parte de la transacción. Si el trigger hace ROLLBACK, también se cancela la operación original.
  • 2
    Usá siempre SET NOCOUNT ON. Evita mensajes de filas afectadas que pueden confundir a la aplicación cliente.
  • 3
    INSTEAD OF y AFTER no son equivalentes. En INSTEAD OF la fila todavía no está aplicada; en AFTER sí.
  • 4
    Se pueden habilitar o deshabilitar. Útil para mantenimiento o cargas masivas.
  • 5
    Evitá lógica pesada. Cursores, consultas costosas o demasiada lógica de negocio pueden degradar el rendimiento.
habilitar_deshabilitar.sql 
DISABLE TRIGGER tr_Marca ON Marcas;
ENABLE TRIGGER tr_Marca ON Marcas;

DISABLE TRIGGER ALL ON Marcas;

Ventajas

  • Lógica centralizada en la base.
  • Ejecución automática.
  • Muy útiles para auditoría.
  • Permiten validaciones complejas.

Desventajas

  • La lógica queda menos visible.
  • Pueden afectar rendimiento.
  • Pueden encadenarse entre sí.
  • Un error puede cancelar toda la transacción.
