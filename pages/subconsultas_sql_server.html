Subconsultas en
SQL Server

Dominá uno de los conceptos más poderosos del lenguaje SQL: consultas anidadas, correlacionadas y operadores avanzados.

¿Qué es una subconsulta?

Una subconsulta es una sentencia SELECT que se anida dentro de otra consulta. La consulta exterior utiliza el resultado de la interior para realizar su operación.

Pensala así: una subconsulta es una pregunta dentro de otra pregunta.
Primero SQL resuelve la interior, obtiene un resultado, y luego la exterior usa ese resultado para filtrar, comparar o calcular.
[ ]

¿Dónde van?

En SELECT, WHERE o FROM. Cada ubicación tiene un propósito distinto.

{ }

Tipos

Escalares, de lista, correlacionadas. La diferencia está en cómo interactúan con la consulta exterior.

=>

Operadores

IN, EXISTS, ANY, ALL, NOT IN. Cada uno tiene semántica y performance diferente.

⚡

Performance

Cuándo conviene una subconsulta vs un JOIN. El plan de ejecución no miente.

Estructura básica

estructura_basica.sql 
SELECT columna1, columna2
FROM   Tabla
WHERE  columna1 = (
           -- ↑ consulta exterior
           SELECT MAX(alguna_columna)
           FROM   OtraTabla
           -- ↑ subconsulta (consulta interior)
       );

Reglas fundamentales

  • 1
    Siempre va entre paréntesis.
  • 2
    Debe devolver un tipo de dato compatible con el contexto donde se usa.
  • 3
    No puede contener ORDER BY salvo que use TOP.
  • 4
    Las subconsultas en FROM necesitan obligatoriamente un alias.

¿Dónde se puede poner una subconsulta?

Hay tres cláusulas que admiten subconsultas, y cada una tiene un comportamiento distinto.

① En la cláusula WHERE — la más común

Se usa para filtrar filas en función de un resultado calculado. Es el caso más frecuente.

where_ejemplo.sql — empleados que ganan más que el promedio 
SELECT Nombre, Sueldo
FROM   Empleados
WHERE  Sueldo > (
           SELECT AVG(Sueldo)
           FROM   Empleados
       );

¿Qué hace SQL?

  • 1
    Ejecuta la subconsulta: calcula el promedio de todos los sueldos, por ejemplo: 75.000.
  • 2
    Reemplaza la subconsulta por ese número: WHERE Sueldo > 75000.
  • 3
    Filtra y devuelve solo los empleados que cumplen la condición.

② En la cláusula SELECT — columna calculada

Devuelve un valor adicional por cada fila. Debe ser escalar (un solo valor).

select_ejemplo.sql — cuántos empleados tiene cada departamento 
SELECT
    d.Nombre,
    (
        SELECT COUNT(*)
        FROM   Empleados e
        WHERE  e.IdDepartamento = d.IdDepartamento
    ) AS CantEmpleados
FROM Departamentos d;
⚠️
Esta subconsulta hace referencia a d.IdDepartamento de la consulta exterior. Por eso es una subconsulta correlacionada: se ejecuta una vez por cada fila del resultado.

③ En la cláusula FROM — tabla derivada

La subconsulta actúa como una tabla temporal sobre la que podés seguir filtrando o agrupando. Obligatorio usar alias.

from_ejemplo.sql — departamentos con sueldo promedio mayor a $80.000 
SELECT dep, promedio
FROM (
    SELECT
        IdDepartamento AS dep,
        AVG(Sueldo)    AS promedio
    FROM   Empleados
    GROUP BY IdDepartamento
) AS ResumenSueldos        -- alias obligatorio
WHERE promedio > 80000;
💡
¿Por qué no usar directamente HAVING AVG(Sueldo) > 80000? Funciona igual en este caso, pero la tabla derivada te permite reutilizar el valor calculado en múltiples condiciones sin repetir la función.

Tipos de subconsultas

La clasificación más importante no es dónde van, sino cómo se relacionan con la consulta exterior.

① Escalar — devuelve exactamente 1 valor

escalar.sql 
-- ¿Quién tiene el sueldo más alto?
SELECT Nombre, Sueldo
FROM   Empleados
WHERE  Sueldo = (
           SELECT MAX(Sueldo) FROM Empleados
       );
Si la subconsulta escalar devuelve más de una fila, SQL Server lanza un error.
Si devuelve NULL, la comparación con = falla silenciosamente (devuelve UNKNOWN, no TRUE ni FALSE).

② De lista — devuelve múltiples valores (una columna)

lista.sql 
-- Empleados de departamentos de tecnología
SELECT Nombre
FROM   Empleados
WHERE  IdDepartamento IN (
           SELECT IdDepartamento
           FROM   Departamentos
           WHERE  Nombre IN ('Sistemas', 'Infraestructura')
       );

③ Correlacionada — referencia a la consulta exterior

Es la más poderosa y la más costosa. Se ejecuta una vez por cada fila de la consulta exterior.

correlacionada.sql — empleados que ganan más que el promedio de SU departamento 
SELECT e.Nombre, e.Sueldo, e.IdDepartamento
FROM   Empleados e           -- alias "e" de la consulta exterior
WHERE  e.Sueldo > (
           SELECT AVG(e2.Sueldo)
           FROM   Empleados e2
           WHERE  e2.IdDepartamento = e.IdDepartamento
           --                          ↑ referencia a la exterior
       );

¿Por qué es "correlacionada"?

  • 1
    La consulta exterior toma la primera fila de Empleados (ej: María, Departamento 3).
  • 2
    Ejecuta la subconsulta: calcula el AVG del Departamento 3.
  • 3
    Compara el sueldo de María con ese promedio. Si aplica, la incluye.
  • 4
    Repite para cada fila. Con 10.000 empleados → 10.000 ejecuciones de la subconsulta.
🔴
Con tablas grandes, las subconsultas correlacionadas pueden ser muy lentas. A menudo se pueden reescribir como un JOIN con GROUP BY para mejor performance. Ver pestaña Performance.

Operadores con subconsultas

El operador que uses entre la consulta exterior y la subconsulta define la semántica completa de la comparación.

Operador Significado Performance Descripción
IN ¿está en la lista? Media Verdadero si el valor coincide con algún elemento devuelto por la subconsulta. Cuidado con NULLs.
NOT IN ¿no está en la lista? Baja Si la subconsulta devuelve algún NULL, NOT IN no retornará ninguna fila. ⚠️ Error clásico.
EXISTS ¿existe alguna fila? Alta Corta en cuanto encuentra la primera coincidencia. Más eficiente que IN para subconsultas grandes.
NOT EXISTS ¿no existe ninguna? Alta La alternativa segura a NOT IN. No tiene el problema con NULLs.
= / < / > comparación escalar Media La subconsulta debe devolver exactamente 1 fila. Si devuelve más, SQL Server lanza error.
ANY / SOME ¿mayor/menor que alguno? Media Verdadero si la condición se cumple para AL MENOS UN elemento de la lista.
ALL ¿mayor/menor que todos? Media Verdadero si la condición se cumple para TODOS los elementos de la lista.

La trampa de NOT IN con NULLs

trampa_null.sql — este resultado puede sorprenderte 
-- Si Departamentos tiene un IdDepartamento = NULL...
SELECT Nombre
FROM   Empleados
WHERE  IdDepartamento NOT IN (
           SELECT IdDepartamento FROM Departamentos
       );
-- Resultado: 0 filas ← aunque haya empleados sin departamento

-- La solución: usar NOT EXISTS
SELECT Nombre
FROM   Empleados e
WHERE  NOT EXISTS (
           SELECT 1
           FROM   Departamentos d
           WHERE  d.IdDepartamento = e.IdDepartamento
       );
-- Resultado: correcto siempre
¿Por qué pasa esto?
En SQL, NULL no es igual ni distinto a nada. Cuando NOT IN compara con una lista que contiene NULL, la expresión devuelve UNKNOWN en lugar de TRUE o FALSE, y ninguna fila pasa el filtro.

EXISTS vs IN — cuándo usar cada uno

exists_vs_in.sql 
-- Con IN: SQL evalúa todos los valores de la lista primero
SELECT d.Nombre
FROM   Departamentos d
WHERE  d.IdDepartamento IN (
           SELECT IdDepartamento FROM Empleados
       );

-- Con EXISTS: corta al primer match → más rápido en tablas grandes
SELECT d.Nombre
FROM   Departamentos d
WHERE  EXISTS (
           SELECT 1              -- el 1 es convención, podría ser *
           FROM   Empleados e
           WHERE  e.IdDepartamento = d.IdDepartamento
       );

Performance y alternativas

Las subconsultas no siempre son la mejor opción. Conocer las alternativas te ayuda a elegir la herramienta correcta.

Subconsulta correlacionada vs JOIN

El mismo resultado, distinta estrategia de ejecución:

🐢 Subconsulta correlacionada
SELECT e.Nombre, e.Sueldo
FROM   Empleados e
WHERE  e.Sueldo > (
    SELECT AVG(e2.Sueldo)
    FROM   Empleados e2
    WHERE  e2.IdDept
            = e.IdDept
);
🚀 JOIN con GROUP BY (mejor)
SELECT e.Nombre, e.Sueldo
FROM   Empleados e
JOIN  (SELECT IdDept,
              AVG(Sueldo) prom
       FROM  Empleados
       GROUP BY IdDept) x
  ON  x.IdDept = e.IdDept
WHERE e.Sueldo > x.prom;

¿Por qué el JOIN es más rápido?

  • 1
    El JOIN calcula los promedios por departamento una sola vez.
  • 2
    La subconsulta correlacionada los calcula una vez por fila de Empleados.
  • 3
    Con 50.000 empleados: 1 pasada vs 50.000 pasadas. La diferencia es enorme.

Guía de decisión rápida

SituaciónHerramienta recomendadaPor qué
Una sola comparación escalar Subconsulta escalar Simple, legible, rápida.
¿Existe al menos una fila? EXISTS Corta al primer match, no lee toda la tabla.
Comparar contra una lista pequeña IN Legible y efectivo para listas chicas.
Comparar contra una lista grande JOIN El optimizador maneja mejor los JOINs.
Excluir filas (NOT IN) NOT EXISTS Evita la trampa de NULLs de NOT IN.
💡
Siempre verificá el plan de ejecución con CTRL+M en SSMS. A veces el optimizador convierte automáticamente una subconsulta en un JOIN. Pero no siempre lo hace.
