Cláusula CASE
en SQL Server

Lógica condicional dentro de tus consultas: transformá valores, clasificá datos y construí columnas calculadas sin salir del SELECT.

¿Qué es la cláusula CASE?

CASE es la forma de escribir lógica condicional del tipo SI/ENTONCES/SINO directamente dentro de una sentencia SQL. Funciona como el IF-ELSE de otros lenguajes, pero integrado en cualquier parte de tu consulta.

CASE no es una sentencia, es una expresión. Eso significa que devuelve un valor y podés usarla en cualquier lugar donde pondría una columna o un valor: en SELECT, ORDER BY, WHERE, GROUP BY, e incluso dentro de funciones de agregación.

El equivalente programático

Si venís de otro lenguaje, CASE es el IF-ELSE de SQL:

En un lenguaje de programación
// pseudocódigo
if (sueldo > 100000) {
    categoria = "Alto"
} else if (sueldo > 50000) {
    categoria = "Medio"
} else {
    categoria = "Bajo"
}
En SQL con CASE
CASE
  WHEN sueldo > 100000
    THEN 'Alto'
  WHEN sueldo > 50000
    THEN 'Medio'
  ELSE 'Bajo'
END
💡
CASE siempre termina con END. Es obligatorio. Si falta, SQL Server lanza un error de sintaxis.

Las dos formas de CASE

SQL Server admite dos variantes. La diferencia está en dónde se pone la condición de comparación.

① CASE simple — compara contra valores exactos

Se escribe la expresión a comparar una sola vez, después de CASE. Cada WHEN especifica un valor posible.

CASE expresion
  WHEN valor1 THEN resultado1
  WHEN valor2 THEN resultado2
  WHEN ...
  ELSE resultado_por_defecto  ← opcional, pero recomendado
END
case_simple.sql — traducir un código a texto 
SELECT
    Nombre,
    IdEstado,
        CASE IdEstado
        WHEN 1 THEN 'Activo'
        WHEN 2 THEN 'Suspendido'
        WHEN 3 THEN 'Retirado'
        ELSE     'Desconocido'
    END AS EstadoDescripcion
FROM Empleados;

Cómo lo evalúa SQL Server

  • 1
    Evalúa IdEstado para cada fila.
  • 2
    Recorre los WHEN de arriba hacia abajo y devuelve el THEN del primer WHEN que coincide.
  • 3
    Si ningún WHEN coincide, devuelve el valor del ELSE.
  • 4
    Si no hay ELSE y ningún WHEN coincide, devuelve NULL.

② CASE buscado — condiciones libres con operadores

No hay expresión después de CASE. Cada WHEN tiene su propia condición completa, lo que permite usar >, <, LIKE, BETWEEN, IS NULL, etc.

CASE
  WHEN condicion1 THEN resultado1
  WHEN condicion2 THEN resultado2
  WHEN ...
  ELSE resultado_por_defecto
END
case_buscado.sql — clasificar por rango de sueldo 
SELECT
    Nombre,
    Sueldo,
        CASE
        WHEN Sueldo >= 150000 THEN 'Premium'
        WHEN Sueldo >= 100000 THEN 'Alto'
        WHEN Sueldo >=  50000 THEN 'Medio'
        ELSE                         'Básico'
    END AS Categoria
FROM Empleados;
⚠️
El orden de los WHEN importa. SQL Server devuelve el resultado del primer WHEN verdadero y no evalúa el resto. Si ponés WHEN Sueldo >= 50000 antes que WHEN Sueldo >= 100000, todos los sueldos altos caerán en la categoría equivocada. Siempre definí las condiciones de mayor a menor especificidad.

Diferencias clave

CASE simpleCASE buscado
Expresión Una vez, después de CASE Cada WHEN tiene la suya
Comparación Solo igualdad (=) Cualquier operador: >, <, LIKE, BETWEEN, IS NULL…
Maneja NULL No (WHEN NULL no funciona) Sí (WHEN col IS NULL)
Cuándo usarlo Valores discretos conocidos Rangos, condiciones complejas

Usos prácticos

CASE aparece en muchos escenarios del mundo real. Estos son los más frecuentes.

① Clasificación por rangos — demo interactivo

Modificá el sueldo con el slider y mirá cómo CASE lo clasifica en tiempo real:

Simulador CASE — clasificación de sueldos

SELECT

        WHEN Sueldo >= 150000 THEN 'Premium'

        WHEN Sueldo >=  50000 THEN 'Medio'
        ELSE                         'Básico'

② CASE en ORDER BY — orden personalizado

Podés definir un orden lógico que no sigue el alfabético ni el numérico.

case_order_by.sql — ordenar por prioridad de estado 
SELECT Nombre, Estado
FROM   Tickets
ORDER BY
        CASE Estado
        WHEN 'Crítico'  THEN 1
        WHEN 'Urgente'  THEN 2
        WHEN 'Normal'   THEN 3
        WHEN 'Cerrado'  THEN 4
        ELSE              99
    END,
    FechaCreacion ASC;

③ CASE dentro de funciones de agregación

Una de las combinaciones más poderosas: contar o sumar solo las filas que cumplen cierta condición.

case_con_sum.sql — contar empleados por categoría en una sola pasada 
SELECT
    COUNT(*)                                              AS Total,
        SUM(CASE WHEN Sueldo >= 100000 THEN 1 ELSE 0 END) AS Categoria_Alto,
        SUM(CASE WHEN Sueldo >=  50000
              AND  Sueldo  < 100000 THEN 1 ELSE 0 END) AS Categoria_Medio,
        SUM(CASE WHEN Sueldo  <  50000 THEN 1 ELSE 0 END) AS Categoria_Basico
FROM Empleados;

¿Por qué funciona?

  • 1
    CASE devuelve 1 si la condición se cumple, 0 si no.
  • 2
    SUM suma todos esos 1s → el total de filas que cumplen la condición.
  • 3
    Esto es equivalente a hacer tres COUNT con WHERE diferentes, pero en una sola consulta.

④ Manejo de NULLs con CASE

CASE buscado puede reemplazar valores NULL con un texto legible.

case_null.sql 
SELECT
    Nombre,
    CASE
        WHEN Telefono IS NULL THEN 'Sin teléfono'
        ELSE Telefono
    END AS ContactoTelefono
FROM Empleados;

-- Nota: ISNULL() hace lo mismo en este caso concreto
-- pero CASE es más flexible para lógica compleja
💡
SQL Server también tiene ISNULL(col, 'valor') y COALESCE(col1, col2) para reemplazar NULLs de forma más concisa. Pero cuando necesitás condiciones más complejas que solo "es NULL o no", CASE es la herramienta.

CASE anidado

Podés poner un CASE dentro del THEN o del ELSE de otro CASE. Esto permite lógica de múltiples dimensiones: primero clasificás por una columna, y dentro de cada rama por otra.

CASE anidado es poderoso, pero debe usarse con criterio. Si el anidamiento supera 2 o 3 niveles, la consulta se vuelve difícil de leer y mantener. En esos casos, conviene revisar si la lógica puede expresarse con condiciones AND/OR en un solo CASE buscado.
case_anidado.sql — categoría según departamento y sueldo 
SELECT
    e.Nombre,
    d.NombreDepto,
    e.Sueldo,
    CASE d.NombreDepto
        WHEN 'Sistemas' THEN
                        CASE
                WHEN e.Sueldo >= 120000 THEN 'Tech Senior'
                WHEN e.Sueldo >=  80000 THEN 'Tech Mid'
                ELSE                          'Tech Junior'
            END
        WHEN 'Ventas' THEN
                        CASE
                WHEN e.Sueldo >= 100000 THEN 'Vendedor A'
                ELSE                          'Vendedor B'
            END
        ELSE 'Área general'
    END AS NivelEmpleado
FROM      Empleados   e
JOIN      Departamentos d
    ON  d.IdDepartamento = e.IdDepartamento;

Cómo lo evalúa SQL Server

  • 1
    El CASE exterior evalúa NombreDepto: si es 'Sistemas', entra al primer CASE interior.
  • 2
    El CASE interior evalúa el sueldo dentro del contexto de ese departamento.
  • 3
    Si el departamento es 'Ventas', entra al segundo CASE interior con su propio umbral.
  • 4
    Si el departamento no es ninguno de los anteriores, el ELSE exterior devuelve 'Área general' directamente, sin entrar a ningún CASE interno.

Alternativa: AND en un solo CASE buscado

El mismo resultado, más legible. En vez de anidar, combinás condiciones con AND:

case_con_and.sql — equivalente sin anidamiento 
SELECT
    e.Nombre,
    CASE
        WHEN d.NombreDepto = 'Sistemas' AND e.Sueldo >= 120000
             THEN 'Tech Senior'
        WHEN d.NombreDepto = 'Sistemas' AND e.Sueldo >= 80000
             THEN 'Tech Mid'
        WHEN d.NombreDepto = 'Sistemas'
             THEN 'Tech Junior'
        WHEN d.NombreDepto = 'Ventas'   AND e.Sueldo >= 100000
             THEN 'Vendedor A'
        WHEN d.NombreDepto = 'Ventas'
             THEN 'Vendedor B'
        ELSE 'Área general'
    END AS NivelEmpleado
FROM  Empleados     e
JOIN  Departamentos d
   ON  d.IdDepartamento = e.IdDepartamento;
💡
En la mayoría de los casos, la versión con AND en un solo CASE buscado es preferible al anidamiento: es más fácil de leer, de modificar y de depurar.

Errores comunes

Estos son los problemas que aparecen con más frecuencia al usar CASE. Conocerlos de antemano te ahorra mucho tiempo de debugging.

① Olvidar el END

❌ Error — falta END
SELECT
    Nombre,
    CASE
        WHEN Sueldo > 100000
        THEN 'Alto'
        ELSE 'Normal'
    -- falta END ↑
FROM Empleados;
✓ Correcto
SELECT
    Nombre,
    CASE
        WHEN Sueldo > 100000
        THEN 'Alto'
        ELSE 'Normal'
    END AS Cat    -- ← END
FROM Empleados;

② Orden incorrecto de condiciones

SQL Server devuelve el THEN del primer WHEN verdadero. Si los rangos están mal ordenados, los resultados son incorrectos sin lanzar ningún error.

❌ Mal orden — todos caen en "Medio"
CASE
  WHEN Sueldo >= 50000
       THEN 'Medio'
  WHEN Sueldo >= 100000
       THEN 'Alto'  ← nunca llega
  ELSE 'Básico'
END
✓ Orden correcto — de mayor a menor
CASE
  WHEN Sueldo >= 100000
       THEN 'Alto'   ← primero
  WHEN Sueldo >=  50000
       THEN 'Medio'
  ELSE 'Básico'
END
🔴
Este error es especialmente peligroso porque no genera ningún mensaje de error. La consulta se ejecuta sin problemas pero los datos son incorrectos. Siempre revisá el orden de los WHEN cuando trabajás con rangos numéricos o de fechas.

③ CASE simple no puede detectar NULLs

❌ No funciona con NULL
CASE Telefono
  WHEN NULL THEN 'Sin dato'
  -- nunca se cumple: NULL = NULL
  -- es UNKNOWN, no TRUE
  ELSE Telefono
END
✓ Usar CASE buscado con IS NULL
CASE
  WHEN Telefono IS NULL
       THEN 'Sin dato'
  ELSE Telefono
END

④ Tipos de dato incompatibles entre los THEN

Todos los THEN y el ELSE de un mismo CASE deben devolver el mismo tipo de dato (o tipos compatibles). Si mezclás, SQL Server tira error o convierte implícitamente —a veces con resultados inesperados.

❌ Tipos incompatibles
CASE
  WHEN activo = 1
       THEN 'Activo'   ← VARCHAR
  ELSE 0              ← INT
END
-- Error o conversión implícita
✓ Mismo tipo en todos los THEN
CASE
  WHEN activo = 1
       THEN 'Activo'
  ELSE 'Inactivo'  ← VARCHAR
END

Resumen de errores

Error¿Da error?Solución
Falta END Sí, sintaxis Siempre cerrar con END.
Orden de WHEN No — resultado incorrecto De condición más específica a más general.
WHEN NULL en CASE simple No — simplemente no coincide Usar CASE buscado con IS NULL.
Tipos incompatibles A veces — depende de la conversión Usar el mismo tipo en todos los THEN / ELSE.
Sin ELSE No — devuelve NULL Siempre agregar ELSE para controlar el caso por defecto.
