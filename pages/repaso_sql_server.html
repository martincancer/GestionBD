Temas de repaso
Detalle del tema

INSERT, UPDATE, DELETE y SELECT

Elegí otra card para cambiar el contenido
Las sentencias INSERT, UPDATE, DELETE y SELECT forman la base del trabajo diario con SQL Server: agregan, modifican, eliminan y consultan información.

Qué hace cada una

  • INSERT agrega nuevas filas.
  • UPDATE modifica valores existentes.
  • DELETE elimina filas.
  • SELECT consulta datos.

Cuidado importante

En UPDATE y DELETE, omitir WHERE puede afectar muchas filas al mismo tiempo. Es una de las advertencias más importantes para quienes recién empiezan.

Operaciones básicas
INSERT INTO Paises (id, nombre)
VALUES (1, 'Argentina');

UPDATE Ciudades
SET nombre = 'Lobos'
WHERE Id = 1;

DELETE FROM Paises
WHERE Id = 2;

SELECT id, nombre
FROM productos;
WHERE permite seleccionar solo las filas que cumplen una condición. Es la base para hacer consultas útiles y no traer todos los registros sin criterio.

Operadores de comparación

  • = , < , > para comparaciones simples. igual, menor o mayor
  • != o <> para excluir valores (distinto de)
  • BETWEEN para rangos entre dos valores
  • LIKE para patrones de texto.
  • IS NULL / IS NOT NULL para buscar valores nulos o no.
  • IN para listas fijas.

Utilización

Los operadores de comparación pueden combinarse con operadores lógicos como AND y OR para formar condiciones más complejas. El uso de paréntesis permite agruparlas y controlar su orden de evaluación, como en la operaciones matematicas.

Filtros frecuentes
SELECT nombre
FROM productos
WHERE precio > 15; -- productos con precio mayor a 15

SELECT nombre
FROM productos
WHERE precio BETWEEN 50 AND 150; -- productos con precio entre 100 y 150, incluidos esos valores

SELECT nombre
FROM productos
WHERE nombre LIKE '%ora';-- productos cuyo nombre termine en 'ora', por ejemplo computadora

SELECT nombre
FROM productos
WHERE descripcion IS NULL; -- productos cuya descripcion sea nula

SELECT nombre
FROM productos
WHERE precio IN (100, 200, 300); -- productos cuyos precios sean 100, 200 o 300
Los JOINs sirven para consultar datos de varias tablas relacionadas. Son esenciales cuando la información está normalizada y no vive en una sola tabla.

Tipos principales

  • INNER JOIN: devuelve coincidencias en ambas tablas.
  • LEFT JOIN: conserva todas las filas de la tabla a la izquierda del JOIN.
  • RIGHT JOIN: conserva todas las filas de la tabla a la derecha del JOIN.

Qué mirar siempre

La condición del ON es tan importante como el tipo de join. Si esa condición está mal, el resultado también va a estarlo, incluso si la consulta corre.

INNER y LEFT JOIN
SELECT
  v.fecha,
  c.nombre,
  v.importe
FROM Ventas v
INNER JOIN Clientes c ON c.clienteId = v.clienteId;

SELECT c.nombre, p.nombre
FROM Ciudades c
LEFT JOIN Paises p ON c.pais_id = p.id;
ORDER BY se usa para ordenar el resultado de una consulta según una o más columnas, de forma ascendente o descendente.

Qué permite hacer

  • Ordenar por una columna numérica, de texto, de fecha, etc.
  • Usar ASC o DESC.
  • Si no se indica ASC o DESC el orden predeterminado es ascendente(ASC).

Punto clave

Si no usás ORDER BY, el orden del resultado no está garantizado. Por eso conviene utilizarlo como parte esencial de una consulta que necesita una presentación ordenada.

Orden ascendente y descendente
SELECT nombre
FROM productos
ORDER BY precio ASC;

SELECT nombre
FROM productos
ORDER BY precio DESC;
TOP limita la cantidad de filas que devuelve una consulta. Es útil cuando querés mostrar solo los primeros registros del resultado, o realizar una paginación.

Qué conviene recordar

  • TOP 10 devuelve solo diez filas.
  • WITH TIES agrega empates con la última fila incluida.
  • TOP se debe usar siempre con ORDER BY

Punto clave

Sin un orden definido, no queda claro cuáles son los “primeros” registros. Por eso `TOP` y `ORDER BY` suelen enseñarse juntos aunque sean conceptos distintos.

TOP y WITH TIES
SELECT TOP 10 nombre, precio
FROM Productos
ORDER BY precio;

SELECT TOP 10 WITH TIES nombre, precio
FROM Productos
ORDER BY precio;
UNION combina los resultados de dos consultas compatibles. UNION ALL hace lo mismo, pero sin eliminar duplicados.

Condición necesaria

  • Las consultas deben devolver la misma cantidad de columnas.
  • Los tipos de datos deben ser compatibles.
  • El orden de las columnas debe corresponderse.

Diferencia clave

UNION limpia duplicados; UNION ALL conserva todo. Esa diferencia es útil para comparar datasets o juntar listados de origen distinto.

UNION y UNION ALL
SELECT nombre, edad FROM ciclistas
UNION
SELECT nombre, edad FROM patinadores;

SELECT nombre, edad FROM ciclistas
UNION ALL
SELECT nombre, edad FROM patinadores;
Las funciones de agregación hacen un resumen de los datos. GROUP BY arma grupos y HAVING filtra esos grupos una vez calculados los resultados agregados.
producto categoría
Cuchillo Cocina
Sartén Cocina
Jeans Ropa
Remera Ropa
Smart TV Electrónica
→
categoría count
Cocina 2
Ropa 2
Electrónica 1

Funciones de agregacion

  • AVG(expr) Promedio.
  • COUNT(expr) Recuento.
  • MAX(expr) Máximo.
  • MIN(expr) Mínimo.
  • SUM(expr) Suma.

Diferencia importante

WHERE filtra filas antes de agrupar. HAVING filtra grupos después de calcular la agregación. Esa diferencia suele costar al principio, así que conviene remarcarla.

COUNT, GROUP BY y HAVING
SELECT COUNT(id)
FROM productos;

SELECT categoria, SUM(precio)
FROM productos
GROUP BY categoria;

SELECT categoria, AVG(precio)
FROM producto
GROUP BY categoria
HAVING AVG(precio) > 3.0;
Una vista es una tabla virtual basada en una consulta. No almacena datos propios, sino que muestra resultados obtenidos desde otras tablas de forma organizada.

Para qué sirve

  • Reutilizar consultas frecuentes.
  • Presentar datos ya procesados.
  • Simplificar acceso a consultas más complejas.

Cómo pensarla

Podés verla como una consulta guardada con nombre.

Crear y consultar una vista
-- crear una vista
CREATE VIEW v_PreciosCategoria
AS
SELECT
  c.categoria,
  SUM(p.precio) AS Total
FROM Categorias c
INNER JOIN Productos p ON p.categoriaid = c.categoriaid
GROUP BY c.categoria;

-- consultar una vista
SELECT * FROM v_PreciosCategoria;
Las funciones escalares de usuario reciben uno o más parámetros, ejecutan lógica y devuelven un único valor. Son útiles cuando querés encapsular cálculos reutilizables.

Características

  • Se crean con CREATE FUNCTION.
  • Definen un tipo de retorno con RETURNS.
  • Devuelven un único valor.
  • Se pueden invocar desde una consulta.

Cuándo usarla

Cuando un cálculo se repite y querés centralizarlo y encapsularlo dentro de la base de datos.

Función escalar
CREATE FUNCTION uf_TotalVentasCliente(@ClienteID INT)
RETURNS NUMERIC(10, 2)
AS
BEGIN
  DECLARE @TotalVentas NUMERIC(10, 2);
  SELECT @TotalVentas = SUM(importe)
  FROM Ventas
  WHERE ClienteID = @ClienteID;
  RETURN @TotalVentas;
END;
Un procedimiento almacenado es un conjunto de instrucciones SQL guardadas en la base. Puede recibir parámetros y ejecutar consultas u operaciones de inserción, modificación o eliminación.

Tipos principales

  • Procedimiento de consulta.
  • Procedimiento de inserción.
  • Procedimiento de modificación.
  • Procedimiento de eliminación.

Cómo se ejecutan

Se invocan normalmente con EXEC o EXECUTE, pasando parámetros según el orden o por nombre.

Consulta
-- Creacion
CREATE PROCEDURE sp_SelClientes
(@ciudadid AS int)
AS
BEGIN
  SELECT razonSocial, nroDoc
  FROM Clientes
  WHERE ciudadid = @ciudadid
END;

-- Utilizacion
EXEC sp_SelClientes @ciudadid = 1
Inserción
-- Creacion
CREATE PROCEDURE sp_InsClientes
(@razonsocial AS varchar(30),
 @nrodoc AS varchar(30),
 @ciudadid AS int)
AS
BEGIN
  INSERT INTO Clientes (razonSocial, nroDoc, ciudadid)
  VALUES (@razonsocial, @nrodoc, @ciudadid)
END;

-- Utilizacion
EXEC sp_InsClientes @razonsocial = 'Martin Cancer', @nrodoc = '31111234', @ciudadid =1
Modificación
-- Creacion
CREATE PROCEDURE sp_UpdCliente
(@razonsocial AS varchar(30),
 @nrodoc AS varchar(30),
 @ciudadid AS int,
 @clienteid AS int)
AS
BEGIN
  UPDATE Clientes
  SET razonSocial = @razonsocial,
      nroDoc = @nrodoc,
      ciudadid = @ciudadid
  WHERE clienteid = @clienteid
END;

-- Utilizacion
EXEC sp_UpdCliente @razonsocial = 'Juan Perez', @nrodoc = '21122134', @ciudadid =2,  @clienteid = 1
Eliminación
-- Creacion
CREATE PROCEDURE sp_DelClientes
(@clienteid AS int)
AS
BEGIN
  DELETE FROM Clientes
  WHERE clienteid = @clienteid
END;

-- Utilizacion
EXEC sp_DelClientes @clienteid = 1
Las constraints son reglas que se aplican a las columnas o tablas para proteger la integridad de los datos. Ayudan a evitar registros inválidos o inconsistentes.

Restricciones mas comunes

  • PRIMARY KEY identifica de manera única.
  • FOREIGN KEY mantiene relación entre tablas.
  • NOT NULL obliga a cargar un valor.
  • UNIQUE evita duplicados.
  • CHECK valida una condición.
  • DEFAULT asigna un valor por defecto.

Importancia de las restricciones

Las restricciones tiene un gran valor en la base de datos realicionales ya que protegen reglas del negocio y la calidad de los datos.

PK, FK, NOT NULL y UNIQUE
ALTER TABLE Clientes
ADD CONSTRAINT PK_Clientes PRIMARY KEY (ID);

ALTER TABLE Pedidos
ADD CONSTRAINT FK_Pedidos_Clientes FOREIGN KEY (ClienteID)
REFERENCES Clientes (ID);

ALTER TABLE Pedidos
ALTER COLUMN ClienteID INT NOT NULL;

ALTER TABLE Productos
ADD CONSTRAINT UQ_Productos_Codigo UNIQUE (Codigo);
CHECK y DEFAULT
ALTER TABLE Ventas
ADD CONSTRAINT CHK_Importe CHECK (Importe > 0);

ALTER TABLE Clientes
ADD CONSTRAINT DF_Clientes_FechaAlta
DEFAULT GETDATE() FOR FechaAlta;
Recordatorio: las constraints no son solo sintaxis. Son reglas de negocio aplicadas directamente en la base para evitar datos inválidos desde cualquier aplicación que la use.
